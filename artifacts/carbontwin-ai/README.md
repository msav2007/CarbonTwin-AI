## Problem Statement

Personal carbon footprints are invisible. People make dozens of micro-decisions every day — what to eat, how to commute, where to shop — without any feedback on their environmental impact. Existing carbon calculators give you a number and stop there. They don't explain *why* your footprint is what it is, they don't build a model of *you*, and they don't make reducing emissions feel achievable.

The result: most people either underestimate their impact or feel paralysed by it.

## Solution

CarbonTwin AI solves this by creating a **personalised digital twin** of your carbon footprint. Instead of a generic number, you get:

- A named AI persona (`Terra-Sage`, `Ember-Root`) built from your lifestyle signals
- A precision breakdown across 5 emission categories with a single Carbon Score
- Gemini-powered coaching that speaks to your specific motivation — not a template
- What-if scenarios that let you model changes *before* committing
- 1-, 5-, and 10-year projections that make the long arc of change visible

---

## Why This Solution Is Unique

Most carbon footprint tools are calculators. They take your inputs, return a number, and stop. CarbonTwin AI is different in three fundamental ways:

### 1. Persistent Identity, Not Ephemeral Data
Rather than showing a number that disappears when you close the tab, CarbonTwin creates a **named, persistent digital persona** — your Carbon Twin — that updates as you take action. Identity is a stronger motivational lever than data.

### 2. Deterministic AI Personas at Zero API Cost
The twin name, archetype, traits, and personality are generated by hashing user choices — no API call, no latency, no cost. The hash is deterministic, so the same lifestyle always produces the same twin. This makes the persona feel reliable and personally meaningful, not random.

### 3. Motivation-Aware Coaching
The coaching engine reads the user's stated *motivation* (shrink footprint, lower bills, travel smarter, consume less) and frames every recommendation in that motivational context. Gemini AI enriches this when available, but the rule-based fallback is already personalised — not generic template text.

## AI Features

CarbonTwin uses **Google Gemini 1.5 Flash** for three distinct AI workflows:

### 1. Personalised Coaching (`/dashboard/coach`)
The rule-based coaching engine generates baseline insights. When a Gemini API key is present, it replaces generic cards with a narrative coaching response that references the user's specific category breakdown, motivation, and top recommended action.

### 2. What-If Narratives (`/dashboard/what-if`)
Users can model hypothetical lifestyle changes (e.g. "switch to transit + go vegan"). Gemini generates a narrative response describing the environmental, health, and social impact of that specific combination — not a generic template.

### 3. Twin Summary (Reveal Screen)
The Carbon Twin's opening summary is seeded by the rule-based engine and optionally enriched by Gemini to produce a more nuanced, personalised narrative.

**Graceful degradation:** all three features fall back to rule-based output when the API key is absent or the request fails.

### Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Client-side carbon engine** | No backend required; instant calculation, works offline, no data leaves the browser |
| **Deterministic twin name** | Same inputs → same twin; reproducibility makes the product feel reliable |
| **Zustand + persist (v2)** | Lightweight state with schema migration support; avoids React Context prop-drilling |
| **Gemini as optional enrichment** | Product works at 100% without AI; Gemini adds depth without creating a hard dependency |
| **Wouter over React Router** | 2KB vs 50KB; sufficient for a single-SPA with < 10 routes |

## Technical Innovation

### Deterministic Hash-Based Twin Generation
The Carbon Twin persona is generated by a deterministic integer hash of the user's 8 lifestyle choices. This ensures the same inputs always produce the same twin name and personality — no random seed, no API dependency. The hash function uses a simple djb2-style accumulation over the string representations of all choices, producing a stable 32-bit integer.

```
hashChoices(data) → integer
  → NAME_PREFIXES[hash % 8]  + "-" + NAME_SUFFIXES[(hash >> 3) % 8]
  → PERSONALITIES[topCategory]
```

### Dual-Path Architecture (Rule-Based + AI)
Every AI-powered feature has a complete rule-based fallback path. The coaching engine, what-if narratives, and twin summary all produce meaningful output without Gemini. Gemini is treated as an **enrichment layer** — it deepens the output, never gates it. This architectural decision makes the app deployable without an API key and resilient to API outages.

### Confidence-Weighted Live Preview
The onboarding wizard shows a real-time footprint estimate that updates on every input. Missing fields are filled with category-level defaults rather than zeros, so the preview starts at a realistic global average and converges on the user's actual footprint as they progress. A `confidence` score (0–100%) communicates estimate reliability to the user.

### Schema-Versioned Persistent State
Zustand's persist middleware is configured with a schema version (`v2`) and a `migrate` function that clamps step values and clears stale results. This means users who return to the app after a schema change see a consistent, valid state rather than a crash or silent corruption.

## Carbon Score Methodology

CarbonTwin uses emission factors sourced from peer-reviewed lifecycle assessment research:

| Category | Range (kg CO₂/year) | Source basis |
|----------|---------------------|--------------|
| Transport | 80 (walk) – 3,200 (car) | Per-km emission factors × annual distance estimates |
| Food | 800 (vegan) – 2,800 (meat-heavy) | Dietary lifecycle assessments (Poore & Nemecek 2018) |
| Home Energy | 600 (low) – 2,800 (high) × household multiplier | Grid carbon intensity (IEA 2023) |
| Air Travel | 300 (rare) – 3,500 (frequent) | IPCC aviation radiative forcing factors |
| Shopping | 400 (minimal) – 2,200 (frequent) | Consumer goods embodied carbon estimates |

**Carbon Score formula:** `clamp(100 − annualKg / 100, 0, 100)`

A score of **58** represents the global average (4.2t CO₂). Higher is better.
The **1.5°C target** corresponds to ≈4,200 kg/year per person.

### Testing Strategy

CarbonTwin's testing philosophy prioritises **meaningful assertions over test count** and **real logic over mocks**:

**No mocking the carbon engine.** Tests in `src/lib/carbon/` use real calculation logic with real emission factors. Mocking the calculator would test the mock, not the science.

**All union type variants covered.** Every test fixture exercises a different combination of the 5 transport × 4 diet × 3 energy × 3 household × 3 travel × 3 shopping × 4 motivation variants. This ensures the emission factor lookup tables are exhaustively exercised.

**Type guard testing.** `isOnboardingComplete` is tested with every missing field individually, not just an empty object. This catches regressions where a new required field is added to `OnboardingData` but not to the guard.

**Determinism verification.** The twin generation tests assert that the same inputs always produce the same twin name, avatar code, and personality — verifying the hash function is stable across runs.

**Edge case coverage.** Name validation tests cover empty string, whitespace-only, 1-character, exactly-32-character, and 33-character inputs. Score tests cover zero emissions (score = 100), global average (score = 58), and extreme emitters (score clamped to 0).

## Security Architecture

CarbonTwin is designed with a minimal attack surface by construction:

| Concern | Approach |
|---------|---------|
| **No server** | Core engine is fully client-side — zero server-side attack surface |
| **No auth** | No credentials, sessions, or tokens to steal or escalate |
| **No database** | State lives in `localStorage` under the user's own origin |
| **No user data transmitted** | Onboarding answers never leave the browser (except opt-in Gemini calls) |
| **Input sanitisation** | `sanitizeNameInput()` strips HTML-unsafe characters before display |
| **No `eval()`** | No dynamic code execution anywhere in the source |
| **No `dangerouslySetInnerHTML`** | All user content rendered as text nodes |
| **Gemini responses as text** | AI responses are never injected as HTML |
| **Dependency auditing** | `pnpm audit` run before each release; critical CVEs block deploys |

See [SECURITY.md](../../SECURITY.md) for the full security policy and vulnerability reporting process.

## Performance Strategy

CarbonTwin is optimised for **instant perceived performance** at every step:

| Strategy | Implementation |
|----------|--------------|
| **Offline-first calculation** | Carbon engine runs synchronously in < 1ms — no loading state for core features |
| **Minimal bundle** | Wouter (2KB) instead of React Router (50KB); no react-query |
| **Tailwind v4** | CSS generated at build time; no runtime style injection |
| **Framer Motion selective** | Animations only on the reveal screen and landing hero |
| **Zustand over Context** | Selective subscriptions prevent full-tree re-renders on state updates |
| **Vite bundling** | Tree-shaking removes unused shadcn/ui primitives from the bundle |

The production bundle is a single 521 KB JS chunk + 132 KB CSS. Code-splitting dashboard routes (React.lazy) is a planned optimisation that would cut the initial parse cost by ~40%.

## Scalability

CarbonTwin's client-first architecture scales horizontally without infrastructure:

**Carbon engine:** The calculation is a pure function of the input data. It can be extracted as a standalone npm package or run in a Web Worker without modification. Adding new emission categories requires only: (1) adding the factor to `constants.ts`, (2) adding the field to `OnboardingData`, and (3) TypeScript enforces every call site that needs updating.

**AI enrichment:** Gemini calls are isolated in `src/lib/gemini/client.ts`. Swapping to a different model provider (OpenAI, Anthropic, self-hosted) requires only changes to this one file — no callers need to change.

**Multi-tenancy:** The current localStorage persistence could be replaced with a remote database (Supabase, PlanetScale) by swapping out the Zustand persist storage adapter. The store shape and all consumers would remain unchanged.

**Internationalisation:** Emission factors are in a single `constants.ts` file. Regional factor tables could be introduced by parameterising lookups with a `region` argument — the TypeScript type system would surface every affected call site.
